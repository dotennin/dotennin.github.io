デフォルトで有効になっている Amazon RDS の自動バックアップ機能により、データベースとトランザクションログがバックアップされます。Amazon RDS では DB インスタンスのストレージボリュームのスナップショットを自動的に作成し、個々のデータベースだけではなく、その DB インスタンス全体をバックアップします。このバックアップは、バックアップウィンドウと呼ばれるユーザーが設定可能な 30 分間隔の時間に 1 日 1 回行われます。スケジュールを変更した場合、どのタイミングで変更が有効になりますか。

問題 1 の説明および補足

問題2
パブリックサブネットとプライベートサブネットを持つVPC上に、新規にインスタンスを作成してインターネットゲートウェイを作成してVPC にアタッチしました。
新規に作成したインスタンスにElasticIPアドレスを割り当てましたがインスタンスに接続することができません。
接続するには何を行えばよいかもっとも適したものを一つ選択してください。
A
セキュリティーグループで任意のプロトコルとポートを可能にアウトバウンドルールを設定する必要がある
B
NATインスタンスをプライベートサブネット上で起動する必要がある
C
ネットワークACLのすべてのアウトバウンドトラフィックを許可する必要がある
D
「Destination] を [0.0.0.0/0] に指定したカスタムルートテーブルを作成しインターネットゲートウェイへ適用する必要がある
問題 2 の説明および補足 
カスタムルートテーブルを作成するが正解です。
デフォルトで、サブネットを作成するとVPC のメインのルートテーブルに関連付けられますが、インターネットゲートウェイへの ルートは含まれません。カスタムルートテーブルを作成してから、インターネットゲートウェイのルートをそれをサブネットに関連 付けます。

インターネットへの接続するステップは以下の通りです。

ステップ1:サブネットを作成する
ステップ2:インターネットゲートウェイをアタッチする
ステップ3:カスタムルートテーブルを作成する
ステップ4:セキュリティグループルールを更新する
ステップ5Elastic IP アドレスを追加する

ステップ4:セキュリティグループルールを更新する
　→デフォルト設定ではインターネットへのアウトバウンドトラフィックはすべて許可していますがインバウンドトラフィックは全て許可していないため許可する必要があります。
「アウトバウンドルールを設定する必要がある」はすでに設定されているので不正解です。
問題3
パブリックサブネットとプライベートサブネットを持つ VPC の違いはインターネットゲートウェイ向け（送信先0.0.0.0/0）へのルーティングの有無になります。パブリックサブネットにインスタンスを起動しましたが、インターネットにアクセスすることはできません。設定を確認したところ、インターネットゲートウェイのルーティングは有効になっています。このサブネット内の他のインスタンスはインターネットにアクセスできます。インターネット向けの通信を有効にする解決策で最も適しているものを選択してください。
A
インスタンスを再起動してネットワーク設定を有効にします。
B
新しいセキュリティグループを作成して、いずれかのアウトバウンドを許可します。インスタンスにこのセキュリティグループを設定します。
C
インスタンスに Elastic IP アドレスを設定します。
D
ネットワーク ACL で任意のプロトコルとポートを可能にインバウンドおよびアウトバウンドルールを設定します。
問題 3 の説明および補足 
インターネットゲートウェイ
サブネット内の他のインスタンスはインターネットに接続できるため、ネットワーク ACL の設定に問題ありません。通信は行きと戻りが必要です。インターネットゲートウェイでの通信（インスタンス→インターネット）において、戻りの通信はインスタンスのプライベート IP アドレスではなくパブリック IPv4 アドレスまたは Elastic IP アドレスが必要です。

■補足
VPC のサブネットのインスタンスでインターネットのアクセス（双方向）を有効にするには、以下を実行する必要があります。
・VPC にインターネットゲートウェイをアタッチする。
・サブネットのルートテーブルがインターネットゲートウェイに繋がっていることを確認します。
・サブネットのインスタンスに、グローバルに一意な IP アドレス (パブリック IPv4 アドレス、Elastic IP アドレス、IPv6 アドレス) が割り当てられていることを確認します。
・ネットワークアクセスコントロールとセキュリティグループルールがインスタンス間で関連するトラフィックを許可していることを確認します。
インターネットゲートウェイはインスタンスに代わって 1 対 1 の NAT を論理的に行います。そのため、トラフィックが VPC サブネットから出てインターネットへ向かうとき、返信アドレスフィールドは、インスタンスのプライベート IP アドレスではなくパブリック IPv4 アドレスまたは Elastic IP アドレスに設定されます。逆に、インスタンスのパブリック IPv4 アドレスまたは Elastic IP アドレス宛てのトラフィックは、その送信先アドレスがインスタンスのプライベート IPv4 アドレスに変換されてから、VPC に配信されます。
参考URL:インターネットゲートウェイ

問題 5 の説明および補足 
AWS のリージョンとエンドポイント
アプリケーションのデータレイテンシーを減らすため、ほとんどの Amazon Web Services は、リクエストを行うためのリージョンのエンドポイントを提供しています。 エンドポイントは、ウェブサービスのエントリポイントとなる URL です。たとえば、https://dynamodb.us-west-2.amazonaws.com は、Amazon DynamoDB サービスのエントリポイントです。IAM などの一部のサービスはリージョンをサポートしていないので、そのエンドポイントにリージョンは含まれません。Amazon EC2 などの一部のサービスでは、特定のリージョン (例: https://ec2.amazonaws.com) を含まないエンドポイントを指定できます。この場合、AWS はデフォルトとして us-east-1 のエンドポイントを利用します。
参考URL:AWS のリージョンとエンドポイント
問題6
Amazon RDS for Oracle で利用可能なライセンスオプションの内、自分のライセンス使用 (BYOL)を選択することはできますか。
A
できない
B
できる
問題 6 の説明および補足 
ライセンスとサポート
自分のライセンス使用 (BYOL)を選択できます。
Amazon RDS for Oracle を使用するために利用できるライセンスオプションには 2 つのタイプがあります。

・自分のライセンス使用 (BYOL): このライセンスモデルでは、既存の Oracle データベースのライセンスを使用して Amazon RDS で Oracle のデプロイメントを実行できます。BYOL モデルの DB インスタンスを実行するには、DB インスタンスのクラスと実行する Oracle データベースのエディション用の適切な Oracle データベースのライセンス (Software Update License & Support 付き) を持っている必要があります。また、Oracle が定めるクラウドコンピューティング環境での Oracle データベースソフトウェアのライセンスポリシーに従う必要があります。Amazon EC2 環境における DB インスタンス、および Amazon EC2 用の Oracle のライセンスポリシーはこちらからご覧ください。

・ライセンス込み: "ライセンス込み" サービスモデルでは、Oracle のライセンスを個別に購入する必要はありません。Oracle データベースのソフトウェアは、AWS によってライセンス化されています。"ライセンス込み" 料金には、ソフトウェア、基盤となるハードウェアリソース、Amazon RDS マネジメント機能が含まれています。
参考URL:ライセンスとサポート
問題7
オンプレミス上に構築した VMware vSphere 上の仮想サーバを iSCSI テープデバイスへバックアップしています。バックアップデータは同じデータセンターに保管しているため災害時に紛失する可能性があります。アプリケーションの大幅な修正を加えることなく、データの紛失を防止できる AWS のストレージのサービスを選択してください。
A
Amazon Glacier
B
Amazon EMR
C
Amazon S3
D
AWS Storage Gateway
問題 7 の説明および補足 
AWS Storage Gateway
AWS Storage Gateway を使うとバックアップ製品と親和性が高い iSCSI テープデバイスとして扱えるので、保存先である iSCSI イニシエータの宛先を変更するだけで、AWS へのデータバックアップが可能になります。保存先は Amazon S3 によってバックアップされた仮想テープライブラリと、AmazonGlacier によってバックアップされた仮想テープシェルフになり、AmazonGlacier へ移動することでストレージコストを下げることができます。

■補足
Storage Gateway は、NFS、SMB、iSCSI、iSCSI-VTL を使用して、ローカルの本番アプリケーションやバックアップアプリとシームレスに接続します。このため、アプリケーションに修正を加えることなく、AWS クラウドストレージを導入できます。プロトコル変換とデバイスエミュレーションにより、Amazon S3 上の Storage Gateway で管理されるボリュームのブロックデータにアクセスしたり、ファイルをネイティブの Amazon S3 オブジェクトとして保存したりできます。また、S3 に保存されている仮想テープライブラリ内で仮想テープバックアップをオンラインで保管したり、このバックアップを Amazon Glacier のテープアーカイブ層に移行することもできます。
参考URL:AWS Storage Gateway の特徴
